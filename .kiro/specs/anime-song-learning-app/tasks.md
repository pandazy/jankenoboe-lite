# Implementation Plan

This plan builds a local Python app that reads and writes one SQLite file, `db/datasource.db`, through a handful of small scripts under `scripts/`. The work starts with the directory layout and dev tooling, then the shared helper module (`scripts/_common.py`), then a test harness, then failing property-based tests that pin down the 16 correctness properties. From there the scripts come in dependency order: `query.py`, `data.py`, `learning.py`, `review.py`, `merge_artists.py`, `cleanup.py`, and the three-step AMQ import pipeline (`import_plan.py`, `import_resolve.py`, `add_play_history.py`). Each script task is immediately followed by unit and integration tests. The last task runs the full suite under `coverage.py` and confirms line coverage is at or above 90%.

Every task maps to one or more specific requirement clauses from `requirements.md` (R1-R20) and, where relevant, to one of the 16 correctness properties from the same document. Tasks marked with `*` are optional: they improve developer quality of life but are not required to hit the 90% coverage target.

The deploy target is a restricted Python 3.10+ environment â€” typically a code-execution sandbox, or any host without network access or `pip install`. See `dev-docs/sandbox-packages.md` for a concrete example of what such an environment typically pre-installs. The runtime under `scripts/` uses only the Python standard library. Tests and dev tooling (pytest, coverage.py, ruff, mypy) are dev-time only â€” they are NOT deployed and NOT expected to run in the target environment; `make package` strips them out before zipping. Property tests use stdlib `random` with a fixed seed. Local dev may install anything convenient into a `.venv` via `requirements-dev.txt` + `make setup`.
