# Implementation Plan

This plan adds `scripts/init_db.py` ŌĆö a new runtime Script that creates `App_Root/db/datasource.db` on a clean host and is a safe no-op afterwards ŌĆö plus the supporting plumbing: a byte-committed `scripts/schema.sql` kept in lockstep with `tests/fixtures/schema.sql`, a small `tools/package.py` edit that re-points `_empty_db` at the new runtime file, and one extra workflow step prepended to every skill so Claude never hits the parent spec's `DB_NOT_FOUND` on a fresh deploy. The work lands in strict dependency order: the schema file + Makefile plumbing (Task 1) has to exist before `scripts/init_db.py` can read it at runtime (Task 2), and `scripts/init_db.py` has to exist before any integration or property test can drive it as a subprocess (Tasks 5 and 6). The `tools/package.py` edit (Task 3) is independent of the runtime path but shares the same new file, so it lands alongside. Skill edits (Task 7) and the coverage checkpoint (Task 8) close the loop.

Every sub-task maps to one or more specific I-N clauses from `requirements.md` and to a named section of `design.md`. Sub-tasks marked with `*` are optional: they are either explicit docstring rewrites beyond the bare minimum or a unit-test case whose coverage is already carried by the integration suite. The runtime imports only the Python standard library; `scripts/init_db.py` deliberately does NOT import `scripts/_common.py` (which pulls in `sqlite3` at top level and would defeat the skip-path deferral required by I-5). No new lint, mypy, or docstring constraints are introduced ŌĆö the existing `pyproject.toml` rules apply unchanged.

  - [x] 2.4 Implement the create path inside `_init_or_skip()`: check the pre-create guard ŌĆö if `db.parent.exists() and not db.parent.is_dir()` (I-2.5), raise `_KnownError("INVALID_INPUT", f"db/ exists but is not a directory: {db_dir}", {"path": str(db_dir)})`. Then lazily `import sqlite3` (first and only place the Script takes the `sqlite3` import hit). Call `db.parent.mkdir(parents=True, exist_ok=True)` inside a `try/except OSError` that raises `_KnownError("INTERNAL_ERROR", f"cannot create db/ directory: {e}", {"path": str(db_dir), "errno": e.errno})` on failure (I-2.6). Read the DDL with `_SCHEMA_PATH.read_text(encoding="utf-8")` ŌĆö a `FileNotFoundError` here is caught by the top-level `except Exception` in `__main__` and surfaces as `INTERNAL_ERROR`. Open the connection with `sqlite3.connect(str(db))`, set `conn.isolation_level = None` (mirrors `_common.open_db`), run `conn.execute("BEGIN IMMEDIATE")`, then `conn.executescript(ddl)`, then `conn.execute("COMMIT")`. On `sqlite3.Error` inside the DDL block, roll back (ignore secondary errors), close the connection, call `db.unlink(missing_ok=True)` to remove the half-written file so a retry lands on the create path again (I-2.7), and raise `_KnownError("INTERNAL_ERROR", f"schema DDL failed: {e}", {"sqlite_error": str(e)})`. Wrap the close in its own `try/except sqlite3.Error` `finally` clause since the conn may already be closed in the error branch. On success, call `_maybe_probe_sys_modules()` and `_success({"created": True, "path": str(db.resolve())})`. (I-2.1, I-2.5, I-2.6, I-2.7, I-3.1, I-4.1, I-4.3, I-4.5, design "`scripts/init_db.py` pseudocode ŌĆö create path", design "Half-written DB cleanup")

- Property I-6's introspection hook is the env var `JANKENOBOE_PROBE_SKIP=1`. When set, `scripts/init_db.py` writes a single `__PROBE__ <json-list>` line to stderr just before emitting the success envelope, where the JSON list is `sorted(m for m in sys.modules if m in {"sqlite3", "argparse"})`. When unset (the default, every normal run), the Script emits nothing extra. The probe is on stderr so it doesn't break the stdout-only success-envelope contract (I-3.5), and the env-var key matches the existing `JANKENOBOE_*` convention in `scripts/_common.py`. The probe exists solely to let Property I-6's test inspect `sys.modules` without a wall-clock timing check.
- No new lint, mypy, or docstring constraints are introduced in this spec. The existing `pyproject.toml` rules (ruff line length, mypy light-touch config, pytest testpaths) apply unchanged to the new and edited files.
- `scripts/schema.sql` is a generated-but-committed artifact: it's a byte-for-byte copy of `tests/fixtures/schema.sql` produced by `make schema-sync`, and Task 4.1's byte-equality test enforces that they never drift. The two-file design keeps the developer-facing mental model clean ŌĆö `tests/fixtures/schema.sql` stays the human-readable, git-diffable fixture where schema changes are reviewed; `scripts/schema.sql` is the mechanical runtime copy read by both `scripts/init_db.py` at runtime and `tools/package.py._empty_db` at package time. When a developer edits the fixture by hand (for example to test a migration), they must explicitly run `make schema-sync` to propagate the change to `scripts/`, which is the intended friction ŌĆö a committed `scripts/schema.sql` is a committed contract.
