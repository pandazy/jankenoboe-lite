# Implementation Plan

This plan adds three small artifacts on top of an already-working app: a human-user `README.md` at the repo root, three hand-authored SVG illustrations under `docs/`, and a re-scoping of `docs/` so it ships in the deployable zip alongside a new author-only `dev-docs/` folder. No runtime code changes — every edit is in dev-only territory (`tools/package.py`, the parent-spec docs, the property-test suite, plus the new text artifacts themselves). The runtime under `scripts/` is not touched.

The work lands in strict dependency order so the tree stays runnable after every step. Task 1 moves `docs/sandbox-packages.md` out of the way and updates the parent-spec cross-references first, so nothing new lands in `docs/` until it is clean. Task 2 wires `docs/` into `tools/package.py` (still a no-op at this point because `docs/` is empty). Task 3 extends the existing packaging property test in place to match the new allowlist. Tasks 4-6 author the three SVGs one at a time (no automated tests — the author opens each in a browser to verify). Task 7 writes the README (no automated tests — the author opens the Markdown preview to verify). Task 8 is the final cross-check.

Every sub-task cites the specific R-UR-N clauses and named Property (P-UR-N from requirements.md, shown as Property 1 in design.md — the packaging rule is the only property backed by an automated test; the rest are author-side review) it implements. Sub-tasks marked with `*` are optional — either docstring polish or a secondary sanity check whose coverage is already carried by another task in the plan. Top-level tasks are never optional. Every task that writes code carries its own test evidence. Tests use stdlib `random.Random(seed)` where randomness applies; `hypothesis` is not used. No task touches `db/datasource.db` — the existing `_guard_real_db` fixture in `tests/conftest.py` enforces that invariant for every test in the suite. Commit after each task lands; never push.
